# AGENTS

## Кодировка
- Все текстовые файлы проекта (`.py`, `.qml`, `.md`, `.txt`, `.csv`) сохранять строго в `UTF-8`.
- Не допускать смешения `UTF-8` и `cp1251` в одном файле.
- При правках русскоязычных строк проверять, что в UI нет «кракозябр».
- Не использовать Unicode-экранирование `\uXXXX` для обычных UI-строк: писать русский текст напрямую в UTF-8.
- После скриптовых/массовых правок обязательно делать пост-проверку измененных файлов на «кракозябры» и сразу исправлять кодировку до `UTF-8` без смешения с другими кодировками.
- Любые скриптовые правки делать только с явным указанием кодировки `UTF-8` при чтении и записи; запрещено использовать команды/инструменты записи с неявной кодировкой (например, `Set-Content` без `-Encoding utf8`), чтобы не допускать повторной порчи текста.
- При обновлении GitHub Release/Description/Topics через API отправлять JSON только в безопасной UTF-8 форме (предпочтительно через Python `json.dumps(..., ensure_ascii=True)`), затем проверять результат чтением API, чтобы исключить «кракозябры» и символы `?`.
- При публикации и редактировании релизов на GitHub обязательно выполнять пост-проверку: открыть последний релиз по API и убедиться, что в `body` нет символов `?` вместо кириллицы; при обнаружении сразу перезаписывать описание корректным UTF-8 payload.
- Для операций с GitHub приоритетно использовать `gh` (GitHub CLI), если он доступен в окружении. Если `gh` недоступен или не авторизован, перед выполнением GitHub-операций запрашивать у пользователя токен и использовать API только после подтверждения доступа.

## Архитектура и качество кода
- Следовать модульной архитектуре: разделять UI, бизнес-логику, парсинг/данные и утилиты по отдельным файлам и слоям.
- Соблюдать чистоту кода: избегать дублирования, сохранять понятные имена и короткие, читаемые функции.
- Писать комментарии к функциям на русском языке в кодировке UTF-8 без BOM.
- Не допускать чрезмерно больших файлов с большим объемом программного кода: при росте сложности выносить логику в отдельные модули.
- После добавления изменений запускать тесты/проверки проекта и фиксировать результат.
- Каждый commit оформлять понятным русским сообщением: короткий заголовок + подробное описание простыми словами (что сделано, зачем, и какой практический результат для пользователя/оператора).
- Перед созданием release обязательно: (1) собрать проект через актуальный spec, (2) проверить, что версия билда увеличена, (3) публиковать в релиз не одиночный exe, а полный portable-архив со всеми зависимостями.

## Qt/QML стиль
- Для приложений на PySide6 QML с кастомизацией Controls (например, background и contentItem) обязательно использовать non-native стиль и устанавливать его до создания `QGuiApplication`: `QQuickStyle.setStyle("Fusion")`.
- На Windows задавать DPI-параметр до импорта Qt (`QT_QPA_PLATFORM=windows:dpiawareness=1`), чтобы избегать предупреждения `SetProcessDpiAwarenessContext() failed`.
- Не использовать нативный стиль Windows для кастомизированных Qt Quick Controls, чтобы исключить предупреждения вида `The current style does not support customization...` и частичную потерю UI-оформления.
